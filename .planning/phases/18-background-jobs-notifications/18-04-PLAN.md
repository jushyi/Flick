**1. process_pending_notifications() -- JOBS-07 drain**

This function runs every 30 seconds and drains the pending_notifications table, grouping entries by target_user_id + source_user_id + type and sending batched notifications.

```sql
CREATE OR REPLACE FUNCTION process_pending_notifications()
RETURNS void LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  batch RECORD;
BEGIN
  -- Group pending notifications older than 30 seconds by target+source+type
  FOR batch IN
    SELECT
      target_user_id,
      source_user_id,
      type,
      jsonb_agg(payload ORDER BY created_at) AS payloads,
      COUNT(*) AS entry_count,
      array_agg(id) AS ids
    FROM pending_notifications
    WHERE created_at <= NOW() - INTERVAL '30 seconds'
    GROUP BY target_user_id, source_user_id, type
    LIMIT 100
  LOOP
    -- Determine notification type based on entry type and count
    -- For reactions: send as 'reaction_batch'
    -- For tags: send as 'tag' (single) or 'tag_batch' (multiple)
    PERFORM net.http_post(
      url := current_setting('app.settings.edge_function_url') || '/send-push-notification',
      headers := jsonb_build_object(
        'Content-Type', 'application/json',
        'Authorization', 'Bearer ' || current_setting('app.settings.service_role_key')
      ),
      body := jsonb_build_object(
        'type', CASE
          WHEN batch.type = 'reaction' THEN 'reaction_batch'
          WHEN batch.type = 'tag' AND batch.entry_count > 1 THEN 'tag_batch'
          WHEN batch.type = 'tag' THEN 'tag'
          ELSE batch.type
        END,
        'user_id', batch.target_user_id,
        'data', jsonb_build_object(
          'source_user_id', batch.source_user_id,
          'count', batch.entry_count,
          'payloads', batch.payloads
        )
      ),
      timeout_milliseconds := 5000
    );

    -- Delete processed entries
    DELETE FROM pending_notifications WHERE id = ANY(batch.ids);
  END LOOP;
END;
$$;
```

Create `supabase/functions/check-push-receipts/index.ts` -- an Edge Function called every 5 minutes by pg_cron to check Expo push receipt statuses and clean up invalid tokens.

The function does:

1. Validate Authorization header
2. Query push_receipts table for entries older than 1 minute (give Expo time to process)
   - SELECT id, ticket_id, user_id, push_token FROM push_receipts WHERE created_at < NOW() - INTERVAL '1 minute' LIMIT 1000
3. If no receipts, return early with `{ checked: 0 }`
4. Extract ticket_ids array, call `checkExpoPushReceipts(ticketIds)` from `../_shared/expo-push.ts`
5. For each receipt:
   - If status is 'ok': just delete the push_receipt row
   - If status is 'error' and details.error is 'DeviceNotRegistered': NULL out the push_token on the users table for that user_id, then delete the push_receipt row
   - If status is 'error' with any other error: log it, delete the push_receipt row
6. Return `{ checked: N, invalidated: M }` where M is the count of DeviceNotRegistered tokens

Import shared module: `import { checkExpoPushReceipts } from '../_shared/expo-push.ts'`
Use `import { createClient } from 'npm:@supabase/supabase-js@2'` for Supabase client with service_role.
Handle CORS and OPTIONS preflight.
Process receipts in chunks of 1000 (Expo API limit).
