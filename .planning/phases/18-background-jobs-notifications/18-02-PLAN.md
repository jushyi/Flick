**1. process_darkroom_reveals() -- JOBS-01**
- UPDATE photos SET status = 'revealed', updated_at = NOW() WHERE status = 'developing' AND reveal_at <= NOW() AND deleted_at IS NULL
- After the UPDATE, query DISTINCT user_id from photos WHERE status = 'revealed' AND updated_at >= NOW() - INTERVAL '3 minutes' AND deleted_at IS NULL
- For each user, also count their revealed photos in that window (for the "N photos ready" message)
- For each user, call pg_net to send-push-notification with type 'photo_reveal' and data: { count: N }
- Schedule: `*/2 * * * *` (every 2 minutes)
- LIMIT 500 on the reveal UPDATE to prevent huge batches
- The pg_net call pattern is:
```sql
PERFORM net.http_post(
  url := current_setting('app.settings.edge_function_url') || '/send-push-notification',
  headers := jsonb_build_object(
    'Content-Type', 'application/json',
    'Authorization', 'Bearer ' || current_setting('app.settings.service_role_key')
  ),
  body := jsonb_build_object(
    'type', 'photo_reveal',
    'user_id', the_user_id,
    'data', jsonb_build_object('count', photo_count)
  ),
  timeout_milliseconds := 5000
);
```

**2. process_streak_expiry() -- JOBS-02**
- Phase 1: Send warnings for streaks within 4 hours of expiry that haven't been warned
  - SELECT streaks with expires_at <= NOW() + INTERVAL '4 hours' AND expires_at > NOW() AND warning_sent = FALSE AND day_count > 0
  - JOIN users for display_name of both users
  - LIMIT 200
  - For each: UPDATE streaks SET warning_sent = TRUE, then call pg_net for BOTH user1 and user2 with type 'streak_warning' and data: { other_name: display_name_of_partner }
- Phase 2: Expire streaks past their expiry time
  - UPDATE streaks SET day_count = 0, last_mutual_at = NULL, expires_at = NULL, warning_sent = FALSE, last_snap_at_user1 = NULL, last_snap_at_user2 = NULL WHERE expires_at IS NOT NULL AND expires_at <= NOW() AND day_count > 0
- Schedule: `*/15 * * * *` (every 15 minutes)

Use `CREATE OR REPLACE FUNCTION` for all functions. All functions return void and use `LANGUAGE plpgsql`. Wrap each function in `SECURITY DEFINER` to bypass RLS (these run as cron, not as a user).

Add cron.schedule calls at the end of the file for these two functions:

**3. cleanup_old_notifications() -- JOBS-04**
- DELETE FROM notifications WHERE created_at < NOW() - INTERVAL '30 days'
- DELETE FROM reaction_batches WHERE sent_at IS NOT NULL AND sent_at < NOW() - INTERVAL '7 days'
- Schedule: `0 2 * * *` (daily at 2 AM UTC)

**4. expire_pinned_snap_notifications() -- JOBS-10**
- Find users with pinned_snap_data that is older than 48 hours
  - SELECT id, pinned_snap_data FROM users WHERE pinned_snap_data IS NOT NULL AND (pinned_snap_data->>'created_at')::timestamptz < NOW() - INTERVAL '48 hours'
  - LIMIT 100
- For each: UPDATE users SET pinned_snap_data = NULL, then call pg_net to send-push-notification with type 'cancel_pinned_snap' and data: {} (this sends a silent push to dismiss the Live Activity)
- Schedule: `0 */2 * * *` (every 2 hours)

**5. send_deletion_reminders() -- part of JOBS-05 lifecycle**
- SELECT users WHERE deletion_scheduled_at IS NOT NULL AND deletion_scheduled_at <= NOW() + INTERVAL '3 days' AND deletion_scheduled_at > NOW() + INTERVAL '2 days 23 hours'
- For each: call pg_net to send-push-notification with type 'deletion_reminder'
- Schedule: `0 9 * * *` (daily at 9 AM UTC)
