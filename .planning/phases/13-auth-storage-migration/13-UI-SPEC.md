### New Copy for Silent Migration

The silent migration (Firebase token bridge) happens behind the scenes on app launch. No new UI is shown during success. On failure, the user is redirected to the standard phone verification flow with this context:

| Element | Copy |
|---------|------|
| Migration fallback heading | "Verify Your Phone Number" |
| Migration fallback body | "Please verify your phone number to continue using Flick." |

This reuses the existing PhoneInputScreen layout. No new screen is needed.

## Interaction States

### PhoneInputScreen (rewritten for Supabase)

Visual layout identical. Behavioral changes:

| State | Current (Firebase) | After (Supabase) | Visual Change |
|-------|-------------------|-------------------|---------------|
| Send code | Calls `signInWithPhoneNumber()`, stores ConfirmationResult in ref | Calls `supabase.auth.signInWithOtp({ phone })`, no ref stored | None |
| Loading | Button shows "Sending...", disabled | Same | None |
| Error | Shake animation + red error text | Same | None |
| Success | Navigates to VerificationScreen with `phoneNumber` and `e164` params | Same navigation, same params | None |
| Country picker | Bottom sheet modal with FlatList | Same | None |

### VerificationScreen (rewritten for Supabase)

Visual layout identical. Behavioral changes:

| State | Current (Firebase) | After (Supabase) | Visual Change |
|-------|-------------------|-------------------|---------------|
| Verify code | Calls `confirmation.confirm(code)` from ConfirmationResult ref | Calls `supabase.auth.verifyOtp({ phone, token, type: 'sms' })` using E.164 from nav params | None |
| Resend | Calls `sendVerificationCode()` again, stores new ConfirmationResult | Calls `supabase.auth.signInWithOtp()` again, no ref needed | None |
| Success | Auth state change triggers navigation via AuthContext | Same -- `onAuthStateChange` fires, AuthContext navigates | None |
| Error | Shake animation + red error text below code input | Same | None |
| Timer | 60-second countdown before resend enabled | Same (Supabase OTP also uses 60-second expiry) | None |

### Silent Migration (new behavior, no new UI)

| State | Visual | Duration |
|-------|--------|----------|
| App launch (existing user, Firebase token present) | Normal app loading screen (existing) | 1-3 seconds |
| Migration success | Seamless transition to main app | Invisible |
| Migration failure | Redirect to PhoneInputScreen with fallback copy | Immediate |

### Upload Queue Drain (new behavior, no new UI)

| State | Visual | Notes |
|-------|--------|-------|
| Old queue items draining via Firebase | No visual indicator (background process) | Happens before switching to Supabase backend |
| New uploads via Supabase | Same camera capture flow, same upload progress indicators | Service layer change only |

## Phase-Specific Notes

1. **This is a service-layer migration phase.** The UI contract exists to confirm that visual output does NOT change. The screens are rewritten in TypeScript calling different backend APIs, but every pixel, animation, and interaction pattern stays the same.

2. **WebP format change is invisible to UI.** Photos switch from JPEG to WebP compression during upload, but `expo-image` renders both formats identically. No visual difference.

3. **Public CDN URLs replace signed URLs for regular photos.** This eliminates the "expired URL flash" bug (broken image that re-fetches). Photos load more reliably, but no UI change is needed to support this.

4. **The only new user-facing copy** is the migration fallback message, which reuses the existing PhoneInputScreen layout with slightly different subtitle/description text.
