## FK Dependency Order (Insert Order)

Based on schema FK analysis:

1. **users** — no FK dependencies (but must have Supabase Auth users first)
2. **photos** — depends on users
3. **photo_reactions** — depends on photos + users
4. **photo_tags** — depends on photos + users
5. **viewed_photos** — depends on photos + users
6. **friendships** — depends on users
7. **blocks** — depends on users
8. **reports** — depends on users
9. **conversations** — depends on users
10. **messages** — depends on conversations + users + photos (tagged_photo_id) + messages (reply_to_id)
11. **streaks** — depends on users
12. **comments** — depends on photos + users + comments (parent_id)
13. **comment_likes** — depends on comments + users
14. **albums** — depends on users + photos (cover_photo_id)
15. **album_photos** — depends on albums + photos
16. **support_requests** — depends on users
17. **message_deletions** — depends on messages + users

**Self-referencing FK challenge:**
- `messages.reply_to_id` → `messages.id`: Insert messages in `created_at` order, earlier messages exist before replies
- `comments.parent_id` → `comments.id`: Insert comments in `created_at` order for same reason

## Storage URL Rewriting (D-04, D-12)

**Firebase Storage URL format:**
```
https://firebasestorage.googleapis.com/v0/b/re-lapse-fa89b.firebasestorage.app/o/photos%2F{userId}%2F{photoId}.jpg?alt=media&token={token}
```

**Supabase Storage path format:**
```
{userId}/{photoId}.jpg
```

**Fields requiring rewrite:**
- `users.profile_photo_path` — Firebase: `profile-photos/{uid}/{filename}` → Supabase: `{supabaseUuid}/{filename}` in `profile-photos` bucket
- `photos.image_url` — Firebase download URL → Supabase public URL: `{SUPABASE_URL}/storage/v1/object/public/photos/{supabaseUuid}/{filename}`
- `photos.storage_path` — Firebase path `photos/{uid}/{id}.jpg` → `{supabaseUuid}/{id}.jpg`
- `messages.snap_storage_path` — Firebase path → Supabase path in `snaps` bucket

**Mapping function pattern (from existing `migrate-firebase-storage.ts`):**
- Firebase path: `photos/{firebaseUid}/{filename}` → Supabase: `{supabaseUuid}/{filename}`
- Firebase path: `profile-photos/{firebaseUid}/{filename}` → Supabase: `{supabaseUuid}/{filename}`
- Firebase path: `snaps/{firebaseUid}/{filename}` → Supabase: `{supabaseUuid}/{filename}`

Key: The Firebase UID in the storage path must be replaced with the Supabase UUID.

## Firestore Timestamp Handling

Firestore uses `Timestamp` objects with `.seconds` and `.nanoseconds`:
```typescript
// Firestore Timestamp → JavaScript Date → ISO string
const isoString = firestoreTimestamp.toDate().toISOString();
```

For Supabase PostgreSQL TIMESTAMPTZ columns, pass ISO 8601 strings directly.

## ID Mapping Strategy

Need a runtime map: `Map<string, string>` for each entity type:
- `userIdMap`: Firebase UID → Supabase UUID (built from users table `firebase_uid` column after auth migration)
- `photoIdMap`: Firestore photo doc ID → new UUID
- `conversationIdMap`: Firestore conversation doc ID → new UUID
- `messageIdMap`: Firestore message doc ID → new UUID (needed for reply_to_id)
- `commentIdMap`: Firestore comment doc ID → new UUID (needed for parent_id)
- `albumIdMap`: Firestore album doc ID → new UUID

## Batch Insert Strategy

Use Supabase's batch insert via `.insert([...rows])`:
- Max recommended batch size: 500-1000 rows per insert
- Use `{ onConflict: 'id' }` or upsert for safe re-runs
- Service role client bypasses RLS

## Validation Architecture

### Count Verification
For each migrated collection:
1. Count Firestore docs: `getCountFromServer(collection(db, name))`
2. Count Supabase rows: `supabase.from(table).select('*', { count: 'exact', head: true })`
3. Compare (accounting for skipped ephemeral collections)

### Per-User Deep Verification (D-15)
1. Look up user by Firebase UID
2. Query all related data in both DBs
3. Compare field-by-field with diff output
4. Check: photo count, friend count, conversation count, message count, album count

### Cross-DB Sampling
- Random sample 10 records per table
- Field-by-field comparison with type coercion (Timestamp → ISO string)
- Flag mismatches with full context

## Risk Areas

1. **Self-referencing FKs** (messages.reply_to_id, comments.parent_id): Must insert in chronological order
2. **Pair tables with CHECK constraints** (friendships, conversations, streaks): Must sort user IDs before insert
3. **Storage URL rewriting**: Must match the paths created by `migrate-firebase-storage.ts`
4. **Empty collections**: Dev Firestore may have empty collections — script must handle gracefully
5. **Large collections**: Messages could be very large — batch processing essential
6. **Firestore subcollection iteration**: Need `collectionGroup` queries or per-parent iteration
7. **JSONB fields**: `selects`, `song`, `pinned_snap_data` — Firestore objects map directly to JSONB
