# Phase 20.1: Firestore Data Migration Script - Context

**Gathered:** 2026-03-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Build a Firestore-to-Supabase data migration script that reads all Firestore collections and writes the data into the corresponding Supabase PostgreSQL tables. Includes an orchestrator script for the full reset-migrate-verify cycle, cross-database verification with per-user deep checks, and a Firebase MCP server setup for interactive Claude-assisted comparison. The script must be repeatable, resumable, and work for both dev (iterative) and prod (one-shot maintenance window) migration.

</domain>

<decisions>
## Implementation Decisions

### Migration Scope & Collection Mapping
- **D-01:** Migrate all persistent Firestore collections: users, photos, friendships, conversations, messages, comments, albums, album_photos, blocks, reports, streaks, photo_reactions, photo_tags, viewed_photos, comment_likes, message_deletions, support_requests.
- **D-02:** Skip ephemeral data: notifications, reaction_batches, pending_notifications, push_receipts. These regenerate naturally in the new system.
- **D-03:** Dev Firestore has partial data coverage — some collections (reports, blocks) may be empty. Script must handle empty collections gracefully (log "0 docs found", skip, continue).
- **D-04:** Rewrite Firebase Storage URLs to Supabase Storage paths during data migration. The Supabase schema uses storage paths (not full URLs), so photo_url, profile_photo_path, and snap_storage_path fields must be transformed.

### Execution Strategy
- **D-05:** Single orchestrator script that migrates all collections in strict dependency order (users first for FK references, then photos, friendships, conversations, messages, etc.). Not per-collection scripts.
- **D-06:** Error handling: skip and log on row-level errors. Log full context (collection, doc ID, error message), skip the row, continue. Print summary of all skipped rows at the end.
- **D-07:** Resume capability via progress JSON file (like migrate-firebase-storage.ts uses .migration-progress.json). Track per-collection progress. If interrupted, pick up where left off.
- **D-08:** Strict FK dependency ordering — insert in dependency order, do not temporarily disable FK constraints. This catches data integrity issues immediately.

### ID Mapping & Foreign Keys
- **D-09:** Auth migration runs FIRST. The migrate-firebase-auth Edge Function creates Supabase Auth users with real UUIDs. The data migration script then maps Firebase UID → Supabase UUID via the `firebase_uid` column on the users table for all FK references.
- **D-10:** Firebase UIDs are 28-char base64 strings (not valid UUIDs), so they cannot be used directly as users.id. The Supabase Auth UUID is the canonical user ID.
- **D-11:** For non-user IDs (photos, comments, messages, etc.): Claude's discretion — check column types in the schema and generate UUIDs where the column is UUID type, preserve strings where it's TEXT.

### Storage URL Rewriting
- **D-12:** Transform Firebase Storage URLs (e.g., `https://firebasestorage.googleapis.com/v0/b/...`) to Supabase Storage paths (e.g., `userId/photoId.jpg`) during data insertion. The storage migration script already copies files to the correct Supabase paths — the data script needs to match those paths.

### Progress Reporting
- **D-13:** Rich console output: per-collection progress (X/Y docs migrated), timing per collection, color-coded success/skip/error counts, final summary table with totals. Console only, no separate log file.

### Verification & Iterative Dev Workflow
- **D-14:** Cross-database comparison: the verification script connects to BOTH Firebase and Supabase simultaneously. Compares total doc counts per collection vs row counts per table, samples random records for field-by-field diffs, flags any missing or mismatched data.
- **D-15:** Per-user deep verification: a `--user <firebase_uid>` flag that traces one user's entire data graph (photos, friendships, conversations, messages, albums, etc.) across both DBs and diffs every record. Essential for debugging migration issues for specific users.
- **D-16:** Orchestrator script (`run-migration-cycle.ts`) runs the full cycle: reset → migrate-auth → migrate-data → migrate-storage → validate. Stops on first verification failure so the user can investigate and fix before rerunning.
- **D-17:** Set up Firebase/Firestore MCP server for Claude Code so Claude can interactively query both Firebase (via MCP) and Supabase (existing MCP) for manual deep-dive comparisons during the iterative dev cycle.

### Dev vs Prod Workflow
- **D-18:** Firebase credentials via service account JSON file path (FIREBASE_SERVICE_ACCOUNT_PATH env var). Different files for dev vs prod.
- **D-19:** Separate .env files: `.env.dev` and `.env.prod` with different Firebase + Supabase credentials. Script accepts `--env dev` or `--env prod` flag to pick which file to load.
- **D-20:** Dry-run mode via `--dry-run` flag: reads all Firestore data, logs collection counts and sample transforms, writes nothing to Supabase. Critical for validating before prod migration.
- **D-21:** Prod migration runs locally from the user's machine (same scripts, `.env.prod` file). Full control, real-time progress monitoring.
- **D-22:** App goes into maintenance mode during prod migration. No new data coming in — migration is a point-in-time snapshot. No need for change data capture or dual-write complexity.

### Prod User Experience
- **D-23:** Seamless auth with fallback: the migrate-firebase-auth Edge Function handles silent auth migration for most users. If it fails for specific users, show a gentle re-verification flow rather than breaking their experience.

### Batch Auth Migration (Added During UAT)
- **D-24:** The per-user Edge Function (`migrate-firebase-auth`) only runs when a user logs in — it cannot batch-migrate all users. A new `scripts/migrate-firebase-auth-batch.ts` was created to iterate all Firebase Auth users and create Supabase Auth users + `firebase_uid` mappings in bulk. This MUST run before the data migration script.
- **D-25:** The orchestrator (`run-migration-cycle.ts`) was updated to run the batch auth script automatically as step 2 (replacing the manual confirmation prompt).
- **D-26:** The reset script (`reset-dev-supabase.ts`) was updated to clean all three progress files: `.migration-progress.json`, `.auth-migration-progress.json`, `.data-migration-progress.json`.

### Field Name Fixes (Found During UAT)
- **D-27:** Firestore `reports` collection uses `reportedUserId` (not `reportedId`). The migrator was updated to check both field names.
- **D-28:** Firestore `streaks` collection stores user IDs in a `participants` array (not `user1Id`/`user2Id` fields). The migrator was updated with a participants array fallback, plus `lastSnapBy` and `warning` field lookups.
- **D-29:** Firestore `photos` collection uses `imageURL` (capital URL) for the storage URL. The migrator was updated to check `imageURL` first.

### Prod Migration Relevance
- **D-30:** Reports, streaks, conversations, messages, message_deletions, and comment_likes are dev-only test data or new features shipping with the Supabase update. Prod users have no data in these collections, so migration failures for these are non-blocking for prod.
- **D-31:** The critical prod collections are: users, photos, photo_reactions, photo_tags, viewed_photos, friendships, albums, album_photos. These contain real user data.

### Claude's Discretion
- Firestore collection structure analysis (flat vs subcollections) — infer from existing code
- Non-user ID handling per table (UUID generation vs string preservation) based on schema column types
- Data transformation logic per collection (timestamp formats, nested docs → JSONB, field renames)
- Exact migration dependency ordering based on FK relationships in the schema
- Batch sizing and concurrency for Supabase inserts

</decisions>

- Existing Supabase MCP server — already configured for interactive SQL queries

<specifics>
## Specific Ideas

- User specifically wants the iterative dev cycle to be the **primary focus** — the ability to reset → migrate → verify → fix → repeat until everything is perfect before attempting prod.
- User wants Claude MCP integration for interactive deep-dive comparisons (Firebase MCP server needs setup, Supabase MCP already exists).
- Per-user verification is critical — being able to trace one user's full data tree across both DBs to debug migration issues.
- Prod migration happens during a maintenance window with the app offline — no dual-write or live migration complexity.
- Seamless auth with graceful fallback for existing users — most users should never notice the backend changed.

</specifics>
