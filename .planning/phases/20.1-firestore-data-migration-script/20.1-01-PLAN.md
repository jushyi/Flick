Create `scripts/migrate-firestore-data.ts` with the following structure:

**CLI argument parsing:**
- `--env dev` or `--env prod` flag: loads `.env.dev` or `.env.prod` from project root. Default: `dev`. Falls back to `.env.local` then `.env` if the specific env file doesn't exist.
- `--dry-run` flag: reads all Firestore data, logs collection counts and sample transforms, writes nothing to Supabase (per D-20).
- `--collection <name>` optional flag: migrate only one specific collection (useful for debugging).
- Parse with simple `process.argv` iteration (no external arg parser needed).

**Environment variables required (per D-18):**
- `FIREBASE_SERVICE_ACCOUNT_PATH` - path to Firebase service account JSON file
- `SUPABASE_URL` or `EXPO_PUBLIC_SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`

**Firebase + Supabase initialization:**
- Follow exact pattern from `migrate-firebase-storage.ts` for Firebase Admin SDK init (service account JSON) and Supabase client init (service role key, `persistSession: false`).
- Initialize Firestore via `admin.firestore()`.

**Progress tracking (per D-07):**
- Progress file at `scripts/.data-migration-progress.json`
- Interface: `{ collections: Record<string, { status: 'pending'|'in_progress'|'complete'|'error', migratedCount: number, skippedCount: number, errorCount: number, errors: Array<{docId: string, error: string}> }>, startedAt: string, lastUpdatedAt: string }`
- Load/save functions following `migrate-firebase-storage.ts` pattern.
- Per-collection tracking: if a collection is already `'complete'` in progress, skip it on re-run (resume capability).

**ID mapping (per D-09, D-10, D-11):**
- `buildUserIdMap(supabase)`: Query `SELECT id, firebase_uid FROM users WHERE firebase_uid IS NOT NULL` to build `Map<string, string>` (Firebase UID -> Supabase UUID). This map is built from data populated by the auth migration (Edge Function) that runs before this script.
- `photoIdMap: Map<string, string>` - Firestore photo doc ID -> generated UUID (built during photos migration, used by photo_reactions, photo_tags, viewed_photos, comments, albums).
- Helper: `mapUserId(firebaseUid: string): string` - looks up in userIdMap, throws if not found (strict FK mode per D-08). Log and skip the row via the error handler.
- Helper: `generateUUID(): string` - use `crypto.randomUUID()`.

**Storage URL rewriting (per D-04, D-12):**
- `rewriteStoragePath(firebasePath: string, firebaseUid: string, supabaseUuid: string): string` - Replaces Firebase UID with Supabase UUID in storage paths. E.g., `photos/{firebaseUid}/{filename}` -> `{supabaseUuid}/{filename}`.
- `rewriteFirebaseUrl(firebaseUrl: string, firebaseUid: string, supabaseUuid: string): string` - For full Firebase download URLs, extract the path component and rewrite. If not a Firebase URL, return as-is.
- `buildSupabasePublicUrl(bucket: string, storagePath: string): string` - Constructs `{SUPABASE_URL}/storage/v1/object/public/{bucket}/{storagePath}`.

**Timestamp conversion:**
- `toISOString(firestoreTimestamp: any): string | null` - Handles Firestore Timestamp objects (`.toDate().toISOString()`), Date objects, ISO strings, null/undefined. Returns null for missing values.

**Batch insert helper:**
- `batchInsert(supabase, table: string, rows: any[], batchSize = 500): Promise<{inserted: number, errors: Array<{index: number, error: string}>}>` - Inserts rows in batches. Uses `.insert(batch)` with no upsert (strict FK mode will catch issues). Returns count of inserted and any errors.
- On error per batch, fall back to row-by-row insert for that batch to identify which specific rows fail (per D-06).

**Console output (per D-13):**
- Color-coded via ANSI escape codes: green for success, yellow for skip, red for error.
- Per-collection: `Migrating {collection}... {X}/{Y} docs ({Z} skipped, {W} errors)` with progress updates.
- Final summary table showing all collections with their counts.

**Collection migrators (9 collections, in FK dependency order per D-05, D-08):**

1. **users**: Read all docs from Firestore `users` collection. For each doc:
   - Look up Supabase UUID via `userIdMap` using the Firestore doc ID as firebase_uid.
   - If user not found in userIdMap, log and skip (auth migration didn't create this user).
   - Map fields: `phone`/`phoneNumber` -> `phone`, `username` -> `username`, `displayName` -> `display_name`, `bio` -> `bio`, `profilePhotoPath` -> `profile_photo_path` (rewrite storage path), `nameColor` -> `name_color`, `selects` -> `selects` (JSONB, pass through), `song` -> `song` (JSONB, pass through), `pinnedSnapData` -> `pinned_snap_data` (JSONB, pass through), `friendCount` -> `friend_count`, `dailyPhotoCount` -> `daily_photo_count`, `lastPhotoDate` -> `last_photo_date`, `fcmToken` -> `fcm_token`, `pushToken` -> `push_token`, `profileSetupCompleted` -> `profile_setup_completed`, `readReceiptsEnabled` -> `read_receipts_enabled`, `deletionScheduledAt` -> `deletion_scheduled_at`, `deletionReason` -> `deletion_reason`, `createdAt` -> `created_at`, `updatedAt` -> `updated_at`.
   - Use UPDATE (not INSERT) since the auth migration already created the user row. Use `.update({...}).eq('id', supabaseUuid)`.
   - For `profile_photo_path`: if it contains a Firebase Storage path like `profile-photos/{firebaseUid}/{filename}`, rewrite to `{supabaseUuid}/{filename}`.

2. **photos**: Read all docs from Firestore `photos` collection. For each doc:
   - Generate new UUID, store in `photoIdMap` mapping Firestore doc ID -> new UUID.
   - Map `userId` -> `user_id` via `mapUserId()`.
   - Map `photoURL`/`imageUrl` -> `image_url`: if Firebase Storage URL, rewrite to Supabase public URL. If storage path like `photos/{firebaseUid}/{filename}`, rewrite to `{supabaseUuid}/{filename}` and build full public URL.
   - Map `storagePath` -> `storage_path`: rewrite Firebase path to Supabase path.
   - Map other fields: `thumbnailDataURL` -> `thumbnail_data_url`, `status` -> `status`, `photoState` -> `photo_state`, `mediaType` -> `media_type`, `caption` -> `caption`, `revealAt` -> `reveal_at`, `commentCount` -> `comment_count`, `reactionCount` -> `reaction_count`, `deletedAt` -> `deleted_at`, `createdAt` -> `created_at`.
   - Skip `localUri` (client-side only).
   - Extract inline `reactions` map -> queue for photo_reactions migration.
   - Extract inline `taggedUserIds` array -> queue for photo_tags migration.
   - Insert with explicit `id` field set to the generated UUID.

3. **photo_reactions**: For each photo that had an inline `reactions` map `{userId: emoji}`:
   - Explode map entries into rows: `{id: generateUUID(), photo_id: photoIdMap[firestorePhotoId], user_id: mapUserId(firebaseUid), emoji: emojiValue}`.
   - Use the inline data collected during photos migration (no separate Firestore read needed).

4. **photo_tags**: For each photo that had an inline `taggedUserIds` array:
   - Explode array into rows: `{photo_id: photoIdMap[firestorePhotoId], user_id: mapUserId(firebaseUid)}`.
   - Use inline data from photos migration.

5. **viewed_photos**: Read subcollection `users/{userId}/viewedPhotos` for each user. For each doc:
   - Map parent userId -> `user_id` via `mapUserId()`.
   - Map the subcollection doc ID or `photoId` field -> `photo_id` via `photoIdMap`.
   - Map `viewedAt` -> `viewed_at`.
   - Skip if photo not found in photoIdMap (photo may not have been migrated).

6. **friendships**: Read all docs from Firestore `friendships` collection. For each doc:
   - Map `user1Id`/`userId1` and `user2Id`/`userId2` -> Supabase UUIDs via `mapUserId()`.
   - Sort the two UUIDs so `user1_id < user2_id` (CHECK constraint).
   - Map `initiatedBy` -> `initiated_by` via `mapUserId()`.
   - Map `status` -> `status`.
   - Insert with generated UUID.

7. **blocks**: Read all docs from Firestore `blocks` collection. For each doc:
   - Map `blockerId` -> `blocker_id` via `mapUserId()`.
   - Map `blockedId` -> `blocked_id` via `mapUserId()`.
   - Composite PK (no generated UUID needed).

8. **reports**: Read all docs from Firestore `reports` collection. For each doc:
   - Map `reporterId` -> `reporter_id`, `reportedId` -> `reported_id` via `mapUserId()`.
   - Map `reason` -> `reason`, `details` -> `details`.
   - Insert with generated UUID.

9. **support_requests**: Read Firestore `supportRequests` collection (camelCase). For each doc:
   - Map `userId` -> `user_id` via `mapUserId()`.
   - Map `category` -> `category`, `message` -> `message`.
   - Insert with generated UUID.

**Empty collection handling (per D-03):** If a Firestore collection query returns 0 docs, log `"{collection}: 0 docs found, skipping"` in yellow and mark as complete in progress.

**Error handling (per D-06):** Each row is wrapped in try/catch. On error, log full context (collection, doc ID, error message), increment error count, continue. At the end, print summary of all skipped/errored rows.

**Main function:** Execute collections in strict dependency order (1-9 above per D-05). After all collections, print final summary table with per-collection counts.
