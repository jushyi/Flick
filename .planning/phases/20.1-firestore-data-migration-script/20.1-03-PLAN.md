Create `scripts/verify-migration-cross-db.ts` that connects to BOTH Firebase Firestore and Supabase simultaneously for comparison.

**CLI arguments:**
- `--env dev` or `--env prod`: loads `.env.dev` or `.env.prod`. Default: `dev`.
- `--user <firebase_uid>`: optional flag for per-user deep verification (D-15). When provided, runs ONLY the per-user deep check instead of the full comparison.
- `--sample-size <N>`: number of random records to sample per collection for field-by-field comparison. Default: 10.

**Environment variables (same as migration script):**
- `FIREBASE_SERVICE_ACCOUNT_PATH`
- `SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`

**Initialization:**
- Firebase Admin SDK init (same pattern as storage migration script).
- Supabase client init with service role key.
- Firestore instance via `admin.firestore()`.

**Result tracking:** Use the `CheckResult` pattern from `validate-migration.ts` with `pass()`/`fail()` helpers and a `printSummary()` at the end.

**Count verification (D-14):**
For each migrated collection pair (Firestore collection -> Supabase table):
| Firestore Collection | Supabase Table | Notes |
|---------------------|----------------|-------|
| `users` | `users` | Count users with firebase_uid IS NOT NULL |
| `photos` | `photos` | Direct count |
| Inline from photos | `photo_reactions` | Sum of reactions map entries across all photos |
| Inline from photos | `photo_tags` | Sum of taggedUserIds array lengths |
| Subcollection | `viewed_photos` | Need to iterate subcollections |
| `friendships` | `friendships` | Direct count |
| `blocks` | `blocks` | Direct count |
| `reports` | `reports` | Direct count |
| `supportRequests` | `support_requests` | Direct count |
| `conversations` | `conversations` | Direct count |
| Subcollection | `messages` | Need to iterate subcollections |
| `message_deletions` or inline | `message_deletions` | Direct count |
| `streaks` | `streaks` | Direct count |
| Subcollection | `comments` | Need to iterate subcollections |
| Nested subcollection | `comment_likes` | Need to iterate nested subcollections |
| `albums` | `albums` | Direct count |
| Inline from albums | `album_photos` | Sum of photoIds array lengths |

For Firestore counts, use `getCountFromServer(collection(db, name))` when available, otherwise `.get()` and count the docs.
For Supabase counts, use `supabase.from(table).select('*', { count: 'exact', head: true })`.
Compare counts. Pass if equal, fail if different (showing expected vs actual).

**Sample verification (D-14):**
For each top-level Firestore collection (users, photos, friendships, conversations, albums, streaks, blocks, reports):
1. Get all Firestore doc IDs from the collection.
2. Randomly sample `--sample-size` doc IDs.
3. For each sampled doc:
   a. Read the full Firestore doc.
   b. Look up the corresponding Supabase row (for users: via firebase_uid; for others: need to find by matching key fields like created_at + user mapping since IDs changed).
   c. Compare key fields with type coercion (Firestore Timestamp -> ISO string, Firebase UID -> Supabase UUID).
   d. Report any mismatches with full diff context.

For the sample verification, the simplest approach is:
- For users: query Supabase `WHERE firebase_uid = '{firestoreDocId}'`, compare fields.
- For photos: store a reverse map during count verification, or compare by `user_id + created_at`.
- For conversations: compare by participant pair.
- If exact matching is too complex, log a warning and skip the sample for that collection.

**Per-user deep verification (D-15):**
When `--user <firebase_uid>` is provided:
1. Look up the user in Firestore by doc ID = firebase_uid.
2. Look up the user in Supabase via `WHERE firebase_uid = '{firebase_uid}'`.
3. If not found in either, fail with message.
4. Compare ALL user fields (field-by-field diff with formatted output).
5. Count and compare per-user data:
   - Photos: Firestore query `where('userId', '==', firebaseUid)` vs Supabase `where user_id = supabaseUuid`.
   - Friendships: count where user is user1 or user2 in both DBs.
   - Conversations: count where user is participant in both DBs.
   - Messages sent: count where sender = user in both DBs.
   - Albums: count where user_id = user in both DBs.
   - Comments: count where user_id = user in both DBs.
   - Blocks (as blocker): count in both DBs.
6. For each category, compare counts and report pass/fail.
7. Sample 3 photos from this user: compare full field set between Firestore and Supabase.
8. Sample 3 messages from this user: compare full field set.
9. Print detailed per-user summary.

**Output format:**
- Color-coded console output (ANSI).
- Per-check pass/fail with details.
- Final summary table.
- Exit code 0 if all pass, 1 if any fail.

**1. Create `scripts/run-migration-cycle.ts` (per D-16):**

Full migration cycle orchestrator that runs steps in sequence, stopping on first failure.

CLI arguments:
- `--env dev` or `--env prod`: passed through to child scripts. Default: `dev`.
- `--skip-reset`: skip the reset step (useful when re-running after fixing a specific collection).
- `--skip-auth`: skip the auth migration step (useful when auth is already migrated).
- `--skip-storage`: skip the storage migration step.
- `--skip-data`: skip the data migration step.
- `--dry-run`: passed through to the data migration script.

Steps (in order):
1. **Reset** (dev only): `npx tsx scripts/reset-dev-supabase.ts`
   - Only runs if `--env dev` (NEVER for prod, per D-21).
   - Skip if `--skip-reset`.
2. **Auth migration**: Log instruction for the user to run the auth migration manually (it's an Edge Function invoked per-user from the client, or a batch script the user runs). Print: "Step 2: Auth migration. Ensure all Firebase users have been migrated via the migrate-firebase-auth Edge Function. Press Enter to continue or Ctrl+C to abort."
   - Use `readline` to wait for user confirmation.
   - Skip if `--skip-auth`.
3. **Data migration**: `npx tsx scripts/migrate-firestore-data.ts --env {env}`
   - Pass through `--dry-run` if set.
4. **Storage migration**: `npx tsx scripts/migrate-firebase-storage.ts`
   - Skip if `--skip-storage`.
5. **Verification**: `npx tsx scripts/verify-migration-cross-db.ts --env {env}`

Each step:
- Print step header with step number and name.
- Run via `execSync` with `stdio: 'inherit'` (streams child output to parent terminal).
- Check exit code. If non-zero, print `"FAILED at step {N}: {name}. Fix the issue and re-run with appropriate --skip flags."` and exit with code 1.
- On success, print `"Step {N} complete."` and proceed.

Final output on success: `"Migration cycle complete! All steps passed."` with timing information.

Safety check for prod: If `--env prod`, require explicit confirmation: `"WARNING: You are about to run against PRODUCTION. Type 'yes-prod' to confirm: "`. Abort if not confirmed.
