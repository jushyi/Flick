5. **Conditional rendering for photo vs video:**
   Replace the existing `<Image>` component with:
   ```jsx
   {isVideo && videoURL ? (
     <TouchableOpacity activeOpacity={1} onPress={() => { toggleMute(); if (onPress) onPress(); }}>
       <VideoPlayer
         source={videoURL}
         isMuted={isMuted}
         onToggleMute={toggleMute}
         loop={true}
         autoPlay={true}
         showControls={false}  // Feed cards don't show progress bar, just play
         isVisible={isVisible}
         style={styles.photo}
         contentFit="cover"
       />
     </TouchableOpacity>
   ) : (
     <TouchableOpacity activeOpacity={0.95} onPress={onPress}>
       <Image source={{ uri: imageURL }} style={styles.photo} contentFit="cover" cachePolicy="memory-disk" />
     </TouchableOpacity>
   )}
   ```

   Actually, per the user decision: "Tap to unmute; once unmuted, sound stays on for subsequent videos until user mutes again." So tapping a video card should unmute (or mute if already unmuted). The onPress for opening PhotoDetail should remain separate. Let's keep the existing onPress behavior and add a separate mute toggle:
   - First tap on a muted video unmutes it (does NOT open PhotoDetail)
   - Subsequent taps toggle mute
   - To open PhotoDetail from a video card, the user would use the existing interaction patterns (or we keep onPress for opening and add a mute button overlay)

   **Simpler approach aligned with user decisions:** Tapping the video toggles mute. The existing onPress for navigating to PhotoDetail still works via the card's TouchableOpacity wrapper (the card area below the photo/video, like the user info row). The video itself handles mute on tap. This matches Instagram-style behavior.

   But looking at the existing FeedPhotoCard, the whole photo area is wrapped in a TouchableOpacity that calls onPress. We need to intercept the tap differently for videos:
   - For videos: tap on the video area toggles mute. Long-press or tap on the info row opens PhotoDetail.
   - For photos: tap opens PhotoDetail (existing behavior).

   **Final approach:** For videos, wrap the VideoPlayer in a TouchableOpacity with `onPress={toggleMute}`. Keep the surrounding card structure the same. The video card's onPress (from parent FeedScreen) still navigates to PhotoDetail when the user taps the info row area below the video.

  <action>
Update FeedScreen to manage video autoplay based on viewport visibility and wrap content with VideoMuteProvider:

1. **No VideoMuteProvider import needed** — the provider is at App.js level (Plan 11-02 Task 3). FeedScreen only needs to manage viewport visibility.

2. **Add viewport visibility tracking:**
   The feed uses `Animated.FlatList`. Use `onViewableItemsChanged` to track which items are currently visible:
   ```javascript
   const [visibleItemIds, setVisibleItemIds] = useState(new Set());

   const viewabilityConfig = useRef({
     itemVisiblePercentThreshold: 50, // Item is "visible" when 50% is in viewport
   }).current;

   const onViewableItemsChanged = useRef(({ viewableItems }) => {
     const ids = new Set(viewableItems.map(item => item.key || item.item?.id));
     setVisibleItemIds(ids);
   }).current;
   ```
   Pass these to the FlatList:
   ```jsx
   <Animated.FlatList
     ...
     onViewableItemsChanged={onViewableItemsChanged}
     viewabilityConfig={viewabilityConfig}
   />
   ```

   **IMPORTANT:** `onViewableItemsChanged` and `viewabilityConfig` must be stable references (useRef or defined outside component). FlatList throws if they change between renders.

3. **Pass isVisible prop to FeedPhotoCard:**
   In the FlatList's renderItem, pass the visibility state:
   ```jsx
   <FeedPhotoCard
     photo={item}
     isVisible={visibleItemIds.has(item.id)}
     onPress={...}
     onCommentPress={...}
     onAvatarPress={...}
     currentUserId={...}
   />
   ```

4. **Do NOT wrap FeedScreen with VideoMuteProvider.**
   VideoMuteProvider is already wired at the App.js root level (Plan 11-02 Task 3). FeedScreen and all other screens (PhotoDetail, stories) share the same mute context instance automatically. No per-screen provider wrapping needed.
