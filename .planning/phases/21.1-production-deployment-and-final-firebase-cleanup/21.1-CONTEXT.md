# Phase 21.1: Production Deployment and Final Firebase Cleanup - Context

**Gathered:** 2026-03-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Create the production Supabase project from scratch, deploy all infrastructure (schema, RLS, Edge Functions, pg_cron, storage buckets), fix Twilio SMS delivery for real OTP, run the production data migration during a maintenance window, cut over the live app to Supabase via a new native EAS build, verify prod is working, then delete all Firebase artifacts from the codebase. The app is distributed via TestFlight (iOS) and direct APK (Android) to 20-200 users.

</domain>

<decisions>
## Implementation Decisions

### Prod Supabase Project Setup
- **D-01:** Create a new prod Supabase project from scratch — nothing exists yet. Deploy full PostgreSQL schema, all RLS policies, Edge Functions, pg_cron jobs, and storage buckets.
- **D-02:** Prod Supabase credentials stored as EAS Secrets (`EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_ANON_KEY`). `app.config.js` selects based on `APP_ENV === 'production'`.
- **D-03:** Twilio phone auth setup included in this phase for prod. Must also fix the dev Twilio SMS delivery issue — texts are attempting to send but failing due to Twilio-side configuration (likely verification/registration, messaging service SID, or geographic permissions). The OTP flow code itself works; the problem is on the Twilio config side.

### Cutover Sequencing
- **D-04:** Pre-cutover OTA from main branch: push an OTA update to the current Firebase-backed production app that shows a maintenance screen, blocking all app functionality during migration.
- **D-05:** New native EAS build from `refactor/backend-optimization` branch (not main). Main has diverged significantly. Merge to main happens after prod is verified working.
- **D-06:** Forced update gate: the maintenance OTA includes version detection logic. After migration, it shows a "Please update" screen with:
  - iOS: Deep link to TestFlight page
  - Android: Direct link to download the new APK (hosted externally — GitHub Releases, Google Drive, or similar)
- **D-07:** App distributed via TestFlight (iOS) and direct APK download (Android). No App Store review needed — TestFlight builds are available immediately.
- **D-08:** Maintenance window during a time when users aren't active. App access is blocked during migration so no new data is created on Firebase after the snapshot.
- **D-09:** Cutover order: Set up prod Supabase → Set EAS Secrets → Build new native binary → Push maintenance OTA to current app → Run migration (auth → data → storage → verify) → Upload new build to TestFlight + host APK → Flip maintenance screen to "please update" screen → Users update to new build.

### Rollback Strategy
- **D-10:** Fix forward only — no rollback to Firebase. If issues surface after cutover, they are fixed on Supabase. The maintenance window stays active until everything works.
- **D-11:** Sentry monitoring + manual spot-testing of key flows + user feedback. Phase 21 UAT already covers dev exhaustively; prod verification is lighter — spot-check key flows and watch Sentry.
- **D-12:** Same-day verification window — if key flows work and Sentry is clean after a few hours, proceed to Firebase cleanup.

### Firebase Cleanup (Post-Verification)
- **D-13:** Delete all Firebase config files: `.firebaserc`, `firebase.json`, `google-services-prod.json`, both service account JSON keys.
- **D-14:** Remove `firebase-admin` from package.json.
- **D-15:** Clean up EAS config: remove `GOOGLE_SERVICES_PLIST` references from all build profiles in `eas.json`.
- **D-16:** Remove `supabase/functions/migrate-firebase-auth/` Edge Function (no longer needed post-migration).
- **D-17:** Delete all migration scripts: `scripts/migrate-firebase-storage.ts`, `scripts/reset-dev-supabase.ts`, `scripts/validate-migration.ts`, `scripts/.migration-progress.json`, and any other migration-specific files. Full clean slate.
- **D-18:** After cleanup, ship a second native build (clean, no Firebase artifacts) bundled with a small new feature. The feature itself is outside this phase's scope — this phase just does the cleanup build.

### Claude's Discretion
- Exact Twilio configuration steps needed (messaging service setup, number verification, geographic permissions)
- Prod Supabase project region selection
- Specific pg_cron job scheduling configuration for prod
- APK hosting location for Android distribution
- Maintenance window timing recommendation
- EAS Secret naming conventions matching the existing app.config.js env var pattern

</decisions>
