# 09-17: Stacked Pinned Snaps Live Activity

## Problem

Currently each pinned snap creates a **separate** Live Activity. When multiple friends pin snaps, the user sees multiple disconnected Live Activities on their lock screen. The desired behavior is a **single stacked Live Activity** showing multiple Polaroids piled on top of each other, like a stack of photos on a table.

## Architecture

### Data Model Changes

**`PinnedSnapAttributes.swift`** — Add stack support to ContentState:

```swift
struct PinnedSnapAttributes: ActivityAttributes {
    // Fixed at creation — uses first snap's info
    let activityId: String      // Now acts as stack ID (first snap's ID)
    let deepLinkUrl: String     // Deep link to messages list (not specific convo)

    struct ContentState: Codable, Hashable {
        var stack: [StackEntry]

        struct StackEntry: Codable, Hashable {
            let snapActivityId: String   // Individual snap identifier
            let senderName: String
            let caption: String?
            let conversationId: String
        }
    }

    // Keep senderName/caption for backward compat but they become the "top" snap's info
    let senderName: String
    let caption: String?
}
```

### Flow

**First pinned snap arrives (no existing Live Activity):**
1. Create new Live Activity with single-entry stack in ContentState
2. Save thumbnail to App Group as `{snapActivityId}.jpg`

**Second+ pinned snap arrives (Live Activity already exists):**
1. Find the existing Live Activity
2. Save new thumbnail to App Group
3. Call `activity.update()` with new entry prepended to stack (newest first)
4. Widget re-renders showing stacked Polaroids

**Snap viewed:**
1. Remove that entry from the stack via `activity.update()`
2. If stack is now empty, call `activity.end()` to dismiss
3. Clean up that snap's thumbnail from App Group

### Native Module Changes

**`LiveActivityManager.swift`** — New/modified methods:

```swift
// Updated: checks for existing activity first, updates if found
@objc func startPinnedSnapActivity(_ params: NSDictionary, resolve: ..., reject: ...)

// New: update existing activity's stack
@objc func updatePinnedSnapStack(_ activityId: String, stack: [[String: Any]], resolve: ..., reject: ...)

// Updated: remove entry from stack, end activity if stack empty
@objc func endPinnedSnapActivity(_ snapActivityId: String, resolve: ..., reject: ...)
```

### JS Service Changes

**`liveActivityService.js`:**
- `startPinnedSnapActivity()` — Check if any active Live Activity exists. If yes, update its stack. If no, create new.
- `removePinnedSnap(snapActivityId)` — Remove one snap from the stack, end activity if empty.
- Track active stack state in memory for quick lookup.

**`App.js`:**
- Foreground handler: instead of always creating, check existing and update
- Resume handler: aggregate all missed pinned snaps into one activity

### Widget UI Changes

**`FlickLiveActivityWidget.swift`** — Lock screen layout:

**Single snap (stack.count == 1):**
- Current Polaroid layout (unchanged from 09-15)

**Multiple snaps (stack.count >= 2):**
- Stack of 2-3 Polaroid frames, slightly offset
- Each Polaroid has its own tilt angle (derived from its snapActivityId)
- Newest on top (fully visible), older ones peek out behind
- Offset: each subsequent frame shifted ~4pt right and ~3pt down
- Max visible frames: 3 (even if stack has more)
- Count badge overlay if stack.count > 3 (e.g., "+2")

```
Lock Screen Layout (stacked):

    ┌──────────┐
   ┌┤  photo3  │  ← newest, full visibility
   │├──────────┤
   ││ photo2   │  ← peeking behind
   │└──────────┘
   └────────────┘     "3 pinned snaps"
                       from Alice, Bob +1
```

**Dynamic Island (compact):**
- Leading: app logo (from 09-16 changes)
- Trailing: count if > 1 (e.g., "3 snaps"), else sender name

**Caption rules (user decision: Option A):**
- **Single snap (stack.count == 1):** Show caption next to Polaroid (current layout, no sender name)
- **Stacked mode (stack.count >= 2):** No captions at all. Just the stacked Polaroids + summary count line. Stacked mode is a visual indicator, not a reading surface.

**Research findings (from 09-17-RESEARCH.md):**
- ContentState supports arrays of Codable structs. 4KB limit → cap at ~10 entries (~150-250 bytes each).
- `activity.update()` replaces ContentState atomically — pass full state, not delta.
- NSE **cannot** call `activity.update()`. For killed-app stack updates, use APNS push updates (`event: "update"`) sent to a per-activity push token (distinct from push-to-start token).
- Per-activity push tokens: after creating an activity, observe `activity.pushTokenUpdates` and store server-side for subsequent APNS update/end events.
- expo-modules-core supports `[Record]` parameters — arrays of structs pass cleanly from JS to Swift.
- Widget re-reads App Groups files on each ContentState update (confirmed: new thumbnails picked up).

## Notes

- ContentState must be Codable/Hashable — keep StackEntry simple
- Thumbnail files use `snapActivityId` as filename (not the activity's native ID)
- Maximum stack size capped at 10 to stay under 4KB ContentState limit
- Push-to-start (09-16) creates the initial activity; per-activity push tokens handle subsequent stack updates
- Per-activity push token: observe `activity.pushTokenUpdates` after creation, store in Firestore for server-side APNS updates
- NSE cannot update activities — server must send APNS push update with `event: "update"` for killed-app stack additions
- The stacking behavior makes the "one Live Activity per conversation" model obsolete — it's now "one Live Activity total for all pinned snaps"
- Single snap with caption uses the layout from current 09-15 (caption only, no sender name, Polaroid at ~1/3 left)
