# 09-16: Push-to-Start Live Activities (Killed App Hybrid)

## Problem

When the app is killed, the NSE intercepts the push notification and attempts to create a Live Activity via `Activity.request()`, but this **fails** — Apple restricts `Activity.request()` in NSE extension processes. The only reliable ways to start a Live Activity when the app is killed are:

1. **Push-to-start tokens (iOS 17.2+)** — The app registers a push-to-start token with Apple. The server sends a special APNS push that starts the Live Activity natively, no NSE needed.
2. **Resume fallback (all iOS)** — When the user opens the app, check for missed pinned snaps and start activities then.

The resume fallback (09-14) already works. This plan adds push-to-start for iOS 17.2+ users as the primary mechanism, keeping the resume fallback for older devices.

## Diagnostic Evidence

NSE logs from killed-app test:
```
didReceive_called → userInfo_keys → bodyData_source → bodyData_keys →
pinned_check → pinned_snap_detected → activity_auth → extracted_data →
current_activities → thumbnail_downloading → thumbnail_downloaded →
thumbnail_saved → thumbnail_verify → calling_activity_request →
activity_request_FAILED → post_request_activities → nse_complete
```

The NSE runs fully, downloads the thumbnail, but `Activity.request()` fails in the extension process.

## Architecture

### Push-to-Start Flow (iOS 17.2+)

1. **App startup**: Register for push-to-start tokens via `Activity.pushToStartToken` observation
2. **Store token**: Send the push-to-start token to Firestore (user document or subcollection)
3. **Server-side**: When a pinned snap is sent, the cloud function sends an APNS push-to-start payload directly (not via Expo Push Service — APNS native required)
4. **Apple delivers**: iOS starts the Live Activity automatically, even when the app is killed
5. **NSE**: Still downloads thumbnail to App Group for the widget to display, but no longer attempts `Activity.request()`

### APNS Push-to-Start Payload Format

```json
{
  "aps": {
    "timestamp": 1234567890,
    "event": "start",
    "content-state": {},
    "attributes-type": "PinnedSnapAttributes",
    "attributes": {
      "activityId": "...",
      "senderName": "...",
      "caption": "...",
      "deepLinkUrl": "lapse://messages/..."
    },
    "alert": {
      "title": "Pinned snap from X",
      "body": "Tap to view"
    }
  }
}
```

### Hybrid Strategy

- iOS 17.2+: Push-to-start (instant, works killed)
- iOS 16.2–17.1: NSE thumbnail download + resume fallback on app open
- All versions: Foreground handler in App.js still works for active app

## Tasks

### Task 1: Register Push-to-Start Token in App

**File: `ios/FlickNativeModules/LiveActivityManager.swift`**

Add a method to observe push-to-start tokens:
- Use `Activity<PinnedSnapAttributes>.pushToStartTokenUpdates` (iOS 17.2+)
- Convert token to hex string
- Expose via native module method `observePushToStartToken` that returns/emits the token
- Start observation on app launch

**File: `src/services/liveActivityService.js`**

Add `registerPushToStartToken()` that:
- Calls native `observePushToStartToken`
- Stores the token in Firestore at `users/{uid}/pushToStartToken`

**File: `App.js`**

- Call `registerPushToStartToken()` on authenticated app startup (iOS only, iOS 17.2+ guard)

### Task 2: Server-Side Push-to-Start Delivery

**File: `functions/index.js`** (onNewMessage handler)

When sending a pinned snap notification:
- Check if recipient has a `pushToStartToken` in their user doc
- If yes, send APNS push-to-start payload directly via Firebase Admin `admin.messaging().send()` with raw APNS config
- If no (or iOS < 17.2), fall back to current Expo Push + NSE path

**File: `functions/index.js`** or new utility

Create helper to build the APNS push-to-start payload with proper structure.

### Task 3: NSE — Stop Attempting Activity.request()

**File: `targets/FlickNotificationService/NotificationService.swift`**

- Keep thumbnail download logic (widget still needs it)
- Remove the `Activity.request()` call — it fails in NSE anyway
- NSE becomes thumbnail-downloader only for pinned snaps
- Still suppress the push banner (deliver empty content) since push-to-start handles the Live Activity

### Task 4: Token Refresh Handling

- Push-to-start tokens can change — observe updates and sync to Firestore
- On sign-out, clear the token from Firestore
- On sign-in, re-register

## Notes

- Push-to-start requires sending APNS directly, not through Expo Push Service — Expo doesn't support push-to-start payloads
- Firebase Admin SDK's `admin.messaging().send()` supports raw APNS payloads
- The APNs auth key must be configured in Firebase project settings (should already be for FCM)
- This is a native change (Swift + cloud functions)
